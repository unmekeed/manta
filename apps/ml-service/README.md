# ML Service

Модели машинного обучения платформы (Гл. 6.2, Гл. 13.3.3). Спринт 10 —
бейзлайн **Win Probability** (Гл. 6.2.2): LightGBM (binary) + изотоническая
калибровка поверх сырого выхода.

## Структура

```
src/
├── app.py                  # gRPC-сервер MLService (Predict/PredictStream)
├── gen/                    # стабы из proto/services.proto (make proto-gen)
├── training/
│   ├── dataset.py          # датасет из MatchTimelineFeatures + синтетика
│   └── train_winprob.py    # обучение + калибровка + артефакт
└── predictors/
    └── win_probability.py  # загрузка артефакта, WP-кривая матча (CLI)
models/                     # артефакты (в git не попадают — реестр/S3)
tests/                      # юнит-тесты конвейера + in-process gRPC-тесты
```

## gRPC-инференс (Гл. 3.7)

Контракт — `proto/services.proto` (источник истины, Гл. 13). Реализовано:

- `MLService.Predict(PredictRequest) → PredictResponse` — WP Radiant по
  `FeatureVector.values` (ключи `training.dataset.FEATURES`; отсутствие
  ключа — `INVALID_ARGUMENT`, неизвестная модель — `NOT_FOUND`);
- `MLService.PredictStream(stream FeatureFrame) → stream WinProbability`
  — потоковая WP-кривая (Гл. 6.2.2: пересчёт каждые N секунд).

```bash
make proto-gen          # перегенерировать стабы после правки proto
make ml-serve           # gRPC на :50051 (GRPC_PORT, MODEL_PATH)
```

## Датасет

Строка — снапшот матча в минуту t из `MatchTimelineFeatures`
(`game_time`, `networth_diff`, `xp_diff`, `kills_diff`, `kills_total`),
target — `radiant_win`. Сплит train/valid — **по матчам** (group split):
снапшоты одного матча скоррелированы, разрез по строкам дал бы утечку.

Пока реальных матчей мало, `--synthetic N` дополняет датасет симуляцией
(преимущество по золоту → sigmoid-вероятность победы). Доля синтетики
фиксируется в метаданных артефакта; такая модель — только для smoke.

## Обучающая выборка и эталон

Разделение по `tier` (колонка витрин, прокидывается конвейером из
Data Collector):

- **train/valid** — высокоранговые паблики (`Premium`, Immortal-скобка
  ≈ 6000+ MMR; Valve раздаёт реплеи пабликов примерно до этого уровня,
  выше — только про-игры) текущего патча, источник `opendota-public`;
- **эталон** — матчи про-команд (`Professional`): в train НЕ попадают
  никогда, на них считается `brier_benchmark_pro` каждой версии.

Промоушен-гейт (`evaluate_gate`) пересчитывает ОБЕ модели — кандидата и
текущую production — на одном holdout **актуальных** данных, а не сравнивает
их сохранённые метрики. Иначе гейт залипает: старая версия, обученная на
маленькой «удачной» выборке, хранит льстивый Brier, которого новые версии на
бОльших данных не достигают, — и правильные кандидаты вечно отклоняются
(реально наблюдали: prod с 63 матчей хранил 0.114, а на текущих данных давал
0.164, хуже кандидата 0.153). Holdout: про-эталон (`Professional`), если
про-матчей достаточно, иначе валидационный сплит (кандидат его не видел).
Решение — по разнице Brier с учётом шума: bootstrap по матчам даёт σ, и
кандидат продвигается, если он **не значимо хуже** (Δ ≤ σ); при равенстве в
пределах шума предпочитается свежая версия на бОльших данных.

Текущая production-версия (163 реальных матча патча 7.41: 105 Immortal
train + 58 pro benchmark, синтетики нет):

| Метрика | Значение | Цель спеки |
|---|---|---|
| Brier calibrated (valid, 21 паблик) | 0.174 | ≤ 0.18 |
| **Brier на про-эталоне (58 матчей, 4008 снапшотов)** | **0.148** | — |

Модель, обученная на высокоранговых пабликах, на тир-1 матчи
переносится даже лучше, чем на паблик-валидацию. WP-кривые без
насыщения в 0/1 (артефакт малых данных ушёл вместе с синтетикой).

## Артефакт

`models/win_probability.pkl` (joblib): booster (model_to_string),
калибратор, список фич, метрики, отпечаток датасета, версия. Валидация
совместимости фич выполняется при загрузке предиктора.

## Качество датасета: сдвиг приора и зеркальная аугментация (v0.3.0)

Расследование «больше данных ухудшает эталон» (`make ml-audit`) вскрыло
**сдвиг приора исхода**: в высокоранговых пабликах Radiant выигрывает
**58.3%**, у про-команд — **44.8%** (сдвиг +0.135). Модель, обученная на
пабликах, наследует side-bias и штрафуется на про-эталоне; с ростом
паблик-данных смещение усиливается.

Решение — **зеркальная аугментация** (`dataset.mirror_xy`, по умолчанию
в `train`): к каждому снапшоту добавляется отражение Radiant↔Dire
(разностные фичи меняют знак, метка инвертируется). Win Probability
обязана быть симметрична по сторонам — `WP(s) = 1 − WP(mirror(s))`;
аугментация обнуляет приор стороны (train-приор 0.594 → 0.500).

Эффект на 945 матчах (один сплит, тот же эталон): **0.1273 → 0.1240** —
рост данных больше не ухудшает перенос. Датасет чист (нет матчей
< 15 мин, дублей между tier нет), поэтому фильтры длительности не нужны.

`make ml-audit` печатает сдвиг приора, распределение длительности и
дубли; запускать перед решением о переобучении на новой порции данных.

## Реестр моделей (Гл. 10.6)

`src/registry` — версионированные артефакты в MinIO (bucket `models`):

```
{name}/versions/{semver}-{run_id}/model.pkl + metadata.json
{name}/stages/production.json          # указатель на версию
```

- `train_winprob --push` загружает версию и продвигает её в
  `production` через **гейт по метрике**: promote только если Brier
  calibrated не хуже текущей production-версии; регресс сохраняется
  как версия, но в сервинг не попадает (продвигается вручную —
  `registry.promote`). Откат = promote старой версии.
- Сервер: `MODEL_PATH=registry://win_probability/production` — артефакт
  скачивается из реестра при старте (проверено против живого MinIO:
  вторая, худшая тренировка гейтом не продвинута).
- `.github/workflows/ml-retrain.yml` — еженедельная проверка конвейера
  обучения на синтетике + ручной запуск переобучения.

### MLflow-бэкенд (Фаза 4)

Интерфейс push/promote/resolve с самого начала повторял семантику MLflow
Registry — и теперь MLflow доступен как второй бэкенд без единой правки
вызывающего кода:

```bash
docker compose -f deployments/docker-compose.yml up -d mlflow   # UI: :9600
REGISTRY_BACKEND=mlflow MLFLOW_TRACKING_URI=http://localhost:9600 \
    make ml-auto-train     # или ml-serve — любой потребитель реестра
```

`registry.mlflow_store.MlflowRegistry` (клиент — mlflow-skinny): push =
run в эксперименте `manta-models` (метрики Brier видны в UI) + артефакты
+ ModelVersion; наш semver-run идентификатор — тег `manta_version`;
стейдж = alias реестра (актуальный API, стейджи MLflow deprecated);
resolve — по alias или номеру версии. Артефакты проксируются через
tracking-сервер (`--serve-artifacts`) — S3-креды клиентам не нужны.
Dev-хранилище — sqlite на volume; прод — тот же сервис с postgres-URI.
Проверено вживую: production-артефакт из S3-реестра прошёл цикл
push → promote → resolve в MLflow байт-в-байт, gRPC-сервер поднялся с
`REGISTRY_BACKEND=mlflow`. По умолчанию остаётся S3-бэкенд — миграция
включается одной переменной, откат — её удалением.

## Автономное переобучение (Гл. 10.4)

`training/auto.py` — цикл continuous training: раз в
`RETRAIN_INTERVAL_S` (6 ч) проверяет витрину и переобучает по ЛЮБОМУ из
двух триггеров:

- **объём** — размер витрины изменился на ≥ `RETRAIN_MIN_NEW_MATCHES` (20)
  с прошлого обучения в этом процессе (по модулю — устойчиво к
  сбросу/перестройке витрины; сравнение с production ушло бы в минус и
  застряло);
- **дрейф** (риск R-02 — смена меты) — PSI распределения фич текущей
  витрины против референса production превысил `RETRAIN_PSI_THRESHOLD`
  (0.2 — конвенциональный порог «значимого» PSI). После баланс-патча
  матчей может быть мало, но игра уже другая — ждать полного порога объёма
  опасно. Дрейф срабатывает только при изменившейся витрине (иначе
  переобучение на тех же данных давало бы ту же модель каждый цикл).

Первый прогон при ≥ `RETRAIN_MIN_TOTAL_MATCHES` (50) обучает сразу.
Продвижение решает гейт по про-эталону, поэтому автономная деградация
невозможна: худшая версия сохраняется, но не сервится.

### Дрейф-мониторинг PSI (`training/drift.py`)

При обучении в артефакт и метаданные версии пишется **референс** —
децильные гистограммы каждой фичи по обучающим данным (≈1 КиБ; auto.py
читает его из `stage_metadata`, не скачивая артефакт). Каждая итерация
считает PSI = Σ (p_cur − p_ref)·ln(p_cur/p_ref) по бинам; интерпретация:
< 0.1 — стабильно, 0.1–0.2 — умеренный сдвиг, > 0.2 — значимый дрейф.
Экспорт в Prometheus (`wp_feature_psi{feature=...}`, `wp_psi_max`) и на
плитку дашборда (`make dashboard`) — тренд виден задолго до порога.
Нюанс чувствительности: чисто мультипликативный сдвиг симметричной около
нуля фичи децили поглощают частично (×1.4 по networth_diff на живых
данных ≈ PSI 0.04), а сдвиги среднего — темп игры, длительность, паттерны
убийств, основной сигнал реальных патчей — ловятся уверенно. Версии без
референса (< 0.4.0) дрейф-триггер не активируют.

```bash
make ml-auto-train        # хост-процесс
make stack-up             # ВЕСЬ конвейер в контейнерах (профиль apps):
                          # collector → parser → extractor → reports +
                          # ml-service (gRPC) + ml-autotrain
```

### Telegram-уведомления об обучении

`training/notify.py` шлёт в Telegram сводку при старте авто-обучения и
после каждого переобучения (продвинута/отклонена гейтом, эталонный и
валидационный Brier, размер датасета). Разовая сводка/тест:
`python -m training.notify`.

Секреты — **только из окружения**, в репозиторий не коммитятся:

| Переменная | Назначение |
|------------|------------|
| `TELEGRAM_BOT_TOKEN` | токен бота от [@BotFather](https://t.me/BotFather); без него уведомления просто выключены |
| `TELEGRAM_CHAT_ID`   | id чата назначения; если не задан — определяется из `getUpdates` по последнему написавшему боту (достаточно один раз отправить боту `/start`) |

Для `make stack-up` задайте их в `deployments/.env` (файл в `.gitignore`).
Локально удобно держать env-файл вне репозитория и подхватывать его:
`set -a && source ~/manta-train.env && set +a && make ml-auto-train`.

Живой прогон цикла: авто-обнаружено +90 матчей → переобучение →
эталонный Brier улучшился 0.1478 → **0.1246** → авто-promote;
следующая итерация корректно пропущена (+9 < 20 порога).
Динамика с ростом данных: 163 матча → 0.148; 253 матча → 0.125.

## Позиционная фича (v0.2.0)

`position_advance` — территориальное продвижение: среднее положение
героев вдоль диагонали карты (миграция 005, источник PositionSnapshots,
прокси Map Control Гл. 6.1.3). Плюс монотонные ограничения LightGBM по
преимуществу (домен: WP не убывает по золоту/опыту/убийствам/территории).

Честная абляция на идентичных данных (453 матча, один сплит): эталонный
Brier 0.1290 с фичей vs 0.1284 без — **в пределах шума**: корреляция с
networth_diff 0.74 съедает добавочный сигнал на текущем объёме. Фича
оставлена в конвейере (монотонное ограничение защищает от шума; ценность
ожидается в эпизодах «золото ровное, карта продавлена» при росте данных).
Гейт корректно удержал прежнюю production-версию.

## SHAP-объяснения (`explain/`, Гл. 6.2)

Вклады считает сам LightGBM — `predict(pred_contrib=True)`, точный
TreeSHAP без внешних зависимостей. Вклады в лог-оддсах; изотоническая
калибровка монотонна, поэтому знак и ранжирование корректно переносятся
на калиброванную WP (численная «доля процента» не обещается).

- `explain.winprob_shap` — `contributions()` (сумма строки + bias == сырой
  скор, проверяется тестом), `top_drivers()` — топ-k по |вкладу|.
- gRPC: `PredictStream` заполняет `WinProbability.top_contributions`
  (топ-3 на кадр); report-generator пишет их в точки таймлайна (`drivers`)
  и прикладывает к каждой ошибке `critical_death` (`top_contributions`) —
  видно, что модель «увидела» в состоянии игры в момент ошибки.
- CLI: `PYTHONPATH=src python -m explain.win_probability MATCH_ID` — WP-кривая
  с драйверами каждой минуты.
