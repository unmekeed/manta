## Назначение документа

Настоящий документ является полной технической спецификацией распределённой интеллектуальной
платформы анализа матчей Dota 2. Он объединяет функциональные требования, нефункциональные
требования (NFR), архитектурное описание всех микросервисов, схемы баз данных, математические
модели оценки игры, спецификации ML-моделей, контракты API (REST/gRPC), архитектуру фронтенда,
контур MLOps, модель безопасности, стратегию наблюдаемости, план развёртывания и дорожную карту.

Документ спроектирован как **единый источник истины (Single Source of Truth)** для реализации
системы. Каждая глава самодостаточна и может использоваться как рабочий бриф для соответствующей
команды.

## Легенда диаграмм

Все диаграммы выполнены в нотации **Mermaid** и рендерятся непосредственно на GitHub/GitLab.
Используются следующие типы:

| Тип диаграммы | Назначение | Mermaid-директива |
|---|---|---|
| Контекстная (C4 L1) | Система и внешние акторы | `flowchart` |
| Контейнерная (C4 L2) | Сервисы и хранилища | `flowchart` |
| ER-диаграмма | Реляционная модель данных | `erDiagram` |
| Диаграмма последовательности | Межсервисные сценарии | `sequenceDiagram` |
| Диаграмма компонентов (UML) | Внутренняя структура сервиса | `flowchart` / `classDiagram` |
| Диаграмма состояний | Жизненный цикл сущностей | `stateDiagram-v2` |
| Диаграмма развёртывания | Kubernetes-топология | `flowchart` |
| Диаграмма Ганта | Дорожная карта | `gantt` |

### Цветовая семантика узлов

## Соглашения о требованиях

- **MUST / ОБЯЗАН** — жёсткое требование, отсутствие которого блокирует релиз.
- **SHOULD / СЛЕДУЕТ** — рекомендуемое требование; отклонение требует обоснования в ADR.
- **MAY / МОЖЕТ** — опциональная возможность.

Каждое формализованное требование имеет идентификатор вида `<КАТЕГОРИЯ>-<ПОДКАТЕГОРИЯ>-<NN>`,
например `NFR-PERF-01`, `FR-DRAFT-04`, `SEC-AUTH-02`.
