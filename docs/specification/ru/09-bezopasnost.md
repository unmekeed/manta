# Глава 9. Безопасность, аутентификация и авторизация

## 9.1. Модель безопасности (обзор)

Безопасность платформы построена по принципам **defense-in-depth** и **zero-trust**: каждый слой
имеет независимые механизмы контроля, внутренние сервисы не доверяют сети по умолчанию.

| Слой | Механизм |
|---|---|
| Периметр | WAF, DDoS-защита, TLS 1.3 терминация |
| Аутентификация | JWT (Bearer), OAuth со Steam |
| Авторизация | RBAC + проверка владения ресурсом |
| Транспорт (внутр.) | mTLS через service mesh |
| Данные | шифрование at-rest (AES-256), хеширование паролей |
| Секреты | Vault / Sealed Secrets |
| Аудит | неизменяемый лог действий |

## 9.2. Аутентификация

### 9.2.1. Потоки аутентификации

| Поток | Использование |
|---|---|
| Steam OpenID/OAuth | вход через Steam-аккаунт |
| Email + пароль | классическая регистрация |
| Refresh-токен | продление сессии без повторного входа |
| Service-to-service | mTLS + короткоживущие токены |

### 9.2.2. JWT — структура и политика

| Параметр | Значение |
|---|---|
| Алгоритм подписи | RS256 (асимметричный) |
| TTL access-токена | 15 минут |
| TTL refresh-токена | 30 дней (ротация) |
| Клеймы | `sub`, `role`, `plan`, `iat`, `exp`, `jti` |
| Ревокация | denylist по `jti` в Redis |
| Ротация ключей | JWKS-эндпоинт, kid-версионирование |

```json
{
  "sub": "account-uuid",
  "role": "premium",
  "plan": "premium_monthly",
  "iat": 1752400000,
  "exp": 1752400900,
  "jti": "token-uuid"
}
```

### 9.2.3. Хранение паролей

| Аспект | Реализация |
|---|---|
| Алгоритм | Argon2id (или bcrypt cost ≥ 12) |
| Соль | уникальная на пользователя |
| Политика паролей | минимум 12 символов, проверка по breach-листам |
| Защита от перебора | rate limit + прогрессивная задержка + CAPTCHA |

## 9.3. Авторизация (RBAC)

### 9.3.1. Роли и права

| Роль | Права |
|---|---|
| `anonymous` | публичная мета, лендинг |
| `free` | загрузка (лимит), базовый анализ |
| `premium` | AI Coach, планы, история, расширенные метрики |
| `pro` (B2B) | командная аналитика, экспорт, повышенные лимиты |
| `admin` | управление пользователями, модерация |
| `service` | внутренние вызовы (по mTLS-идентичности) |

### 9.3.2. Матрица доступа (пример)

| Ресурс / Роль | anonymous | free | premium | pro | admin |
|---|---|---|---|---|---|
| `GET /meta/*` | ✅ | ✅ | ✅ | ✅ | ✅ |
| `POST /matches/upload` | ❌ | ✅ (лимит) | ✅ | ✅ | ✅ |
| `GET /players/{id}/training-plan` | ❌ | ❌ | ✅ | ✅ | ✅ |
| Командная аналитика | ❌ | ❌ | ❌ | ✅ | ✅ |
| Админ-панель | ❌ | ❌ | ❌ | ❌ | ✅ |

### 9.3.3. Проверка владения ресурсом

Помимо роли, проверяется владение: игрок может видеть свои приватные разборы, но не чужие
(если владелец не открыл доступ). Реализуется на уровне API Gateway и сервисов данных.

## 9.4. Модель угроз (Threat Model, STRIDE)

| Категория (STRIDE) | Угроза | Митигация |
|---|---|---|
| **S**poofing | Подделка личности | JWT-подпись, mTLS, MFA (опц.) |
| **T**ampering | Изменение данных в транзите | TLS 1.3, mTLS, целостность payload |
| **R**epudiation | Отрицание действий | Неизменяемый аудит-лог с `trace_id` |
| **I**nformation Disclosure | Утечка PII | Шифрование at-rest, минимизация данных |
| **D**enial of Service | Перегрузка | Rate limiting, автоскейлинг, WAF |
| **E**levation of Privilege | Повышение прав | Строгий RBAC, least privilege |

### 9.4.1. Специфичные для домена угрозы

| Угроза | Митигация |
|---|---|
| Загрузка вредоносного «`.dem`» | валидация магии/размера, песочница парсера |
| Скрейпинг/абьюз API | rate limit, аномалия-детекция, API-ключи |
| Инъекция промптов в LLM | санитизация входа, guardrails, изоляция контекста |
| Эксфильтрация данных через RAG | ограничение источников контекста, ACL на Vector DB |

## 9.5. Защита конвейера данных и ML

| Аспект | Мера |
|---|---|
| Изоляция парсера | запуск в sandbox/gVisor, seccomp, без сети |
| Валидация схем | Schema Registry, отклонение несовместимых сообщений |
| Целостность моделей | подпись артефактов, проверка контрольных сумм |
| Защита от отравления данных | аномалия-детекция входных матчей, карантин |
| Секреты в пайплайнах | инъекция из Vault, запрет хардкода |

---

## 9.6. Управление секретами и ключами

| Секрет | Хранилище | Ротация |
|---|---|---|
| JWT-ключи (RSA) | Vault | 90 дней |
| Пароли БД | Vault (dynamic secrets) | по политике |
| API-ключи внешних систем | Vault | по провайдеру |
| TLS-сертификаты | cert-manager / mesh CA | авто (короткий TTL) |
| Ключи шифрования данных | KMS | ежегодно (envelope encryption) |

## 9.7. Соответствие и приватность (GDPR)

| Требование | Реализация |
|---|---|
| Право на удаление | эндпоинт erasure → каскадное удаление PII |
| Право на экспорт | выгрузка данных пользователя (JSON) |
| Минимизация данных | хранение только необходимого |
| Согласие | явный consent на обработку |
| Псевдонимизация | разделение PII и аналитических идентификаторов |
| Хранение | политики retention (см. Гл. 4.6) |

## 9.8. Безопасность инфраструктуры и SDLC

| Область | Практика |
|---|---|
| Образы контейнеров | сканирование (Trivy), distroless, non-root |
| Зависимости | SCA (Dependabot/Snyk), SBOM |
| Код | SAST в CI, secret-scanning |
| Кластер | Network Policies, Pod Security Standards |
| Доступ | least privilege RBAC в Kubernetes, аудит kube-api |
| Реагирование | runbook инцидентов, ротация при компрометации |

### 9.8.1. Требования безопасности (сводка)

| ID | Требование |
|---|---|
| SEC-AUTH-01 | Все внешние соединения — TLS 1.3 |
| SEC-AUTH-02 | Все внутренние вызовы — mTLS |
| SEC-AUTH-03 | Access-токены живут ≤ 15 мин |
| SEC-DATA-01 | PII шифруется at-rest (AES-256) |
| SEC-DATA-02 | Пароли — Argon2id/bcrypt |
| SEC-PIPE-01 | Парсер изолирован в песочнице без сети |
| SEC-COMP-01 | Поддержка GDPR erasure/export |
