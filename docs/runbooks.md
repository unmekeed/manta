# Runbooks инцидентов (D4)

Сценарии — не гипотетические: каждый случился в живом стеке и был
расследован. Формат: симптом → диагностика → причины по частоте → действия.

**Первый шаг любой диагностики — `make doctor`** (спринт 49): проверяет
конвейер по ДАННЫМ (контейнеры, Kafka-топики и консьюмер-группы, свежесть
ReplayEvents/PositionSnapshots/витрины, квота OpenDota, часы WSL vs
контейнеров, применённость миграций) и печатает OK/WARN/FAIL по каждому
пункту. `make recover` сам прогоняет топики+миграции идемпотентно и
заканчивается doctor'ом. Логи сервисов — в `~/manta-logs`
(переопределяется `MANTA_LOG_DIR`; больше НЕ `/tmp`, который гибнет при
рестарте WSL).

---

## 1. Витрина не растёт

**Симптом.** Telegram-уведомления об обучении прекратились; графика
`training_dataset_matches` — полка; все процессы при этом «up»
(`make recover` показывает OK по всем строкам). Авто-алерт: сообщение
«⚠️ витрина не растёт N ч» (auto-train, `DATASET_STALL_ALERT_H`, по
умолчанию 12 ч) и метрика `training_dataset_stalled=1`.

**Диагностика.**

```bash
# 1) когда витрина остановилась
docker exec manta-clickhouse-1 clickhouse-client -q "
SELECT toStartOfHour(computed_at) h, uniqExact(match_id)
FROM manta.MatchTimelineFeatures WHERE computed_at > now()-INTERVAL 36 HOUR
GROUP BY h ORDER BY h"

# 2) квота OpenDota (сброс в 00:00 UTC)
curl -sI https://api.opendota.com/api/health | grep -i remaining

# 3) 429 в логах и метрике
grep -c "429" ~/manta-logs/timeline.log ~/manta-logs/collector.log
curl -s localhost:9108/metrics | grep opendota_rate_limited_total
```

**Причины по частоте.**

1. **Суточный rate limit OpenDota** (реальный случай 2026-07-19: рост шёл
   только 4:00–7:00 — ровно окно от сброса квоты до её исчерпания; тем
   же вечером на другой машине `remaining-day` улетел до **-930** — то
   есть реальный анонимный лимит заметно ниже заявленных ~1660/сутки
   для платного тарифа, burst 60/мин остаётся ориентиром для обеих
   схем). → Дефолты `make recover` бюджетные (~1100/сутки: timeline 10
   матчей/30 мин, pro-timeline 5/час, реплей-пути 1/час; отвергнутые
   кандидаты кэшируются, detail-вызовы цикла ограничены
   `TIMELINE_DETAIL_BUDGET`), но при таком заниженном реальном лимите
   этого может не хватать — следить за `remaining-day` и при
   регулярных 429 снижать `TIMELINE_LIMIT`/`OPENDOTA_LIMIT` дальше.
   При 429 коллектор сам ждёт до 00:00 UTC + запас вместо обычного
   короткого интервала (не жжёт остаток квоты повторными попытками).
   `OPENDOTA_API_KEY` в `MANTA_TRAIN_ENV` снимает вопрос совсем (код
   поддерживает; сайт opendota.com/api-keys периодически лежит).
2. **Умер процесс конвейера** (перезапуск среды убил dockerd и фоны;
   реальные случаи 2026-07-16/18/19). → `make recover` — идемпотентен.
   Подслучай (2026-07-20): рестарт Docker Desktop пересоздал
   Postgres/Kafka, а процессы коллекторов остались живы со СТАРЫМИ
   соединениями — `pgrep` их видит, recover пропускает, но каждый цикл
   падает `psycopg.OperationalError: server closed the connection`.
   Начиная со спринта 47 коллекторы переподключаются к PG сами
   (лог «postgres: соединение умерло — переподключаюсь»); на старом
   коде — `pkill -f "collector --source"` и `make recover`.
3. **ClickHouse после нечистого рестарта** долго разгребает temp-мерджи
   или упёрся в file-descriptor limit (реальный случай 2026-07-19:
   `Too many open files` в logs → зацикленный мердж `system.metric_log`).
   → дождаться Ready for connections; при зацикленном мердже —
   `docker restart manta-clickhouse-1` в тихое окно.
4. **Код обновился, миграция — нет** (реальный случай: similarity падал
   404 на `networth_total`, отсутствующей в схеме). → `make migrate-ch`.

## 2. Гейт всё отклоняет

**Симптом.** Подряд ≥5 «⏸ отклонена гейтом» в Telegram; в реестре версии
копятся, production не двигается.

**Диагностика.** Смотреть текст причины в уведомлении: гейт печатает
`new X vs prod Y (Δ, σ)` и режим сравнения (fresh-holdout / про-эталон).

**Причины.**

1. **Это норма, а не инцидент**, если Δ в пределах ±2σ и знак скачет:
   датасет вырос на малую долю, реальной разницы нет — гейт честно
   отказывает «не лучше». Вмешательство не нужно; продвижение случится
   на осмысленном приросте данных.
2. **Prod «удачно» стоит на старом эталоне** (случай спринта 27, из-за
   которого гейт и переписан): сравнение шло на разных данных.
   Проверить, что обе модели оцениваются на одном holdout
   (лог `evaluate_gate`: «одни данные (N матчей)»).
3. **Систематически хуже после смены фич** — новая фича добавила NaN-строк
   (JSON-путь не даёт позиций). Проверить долю NaN в
   `training.dataset` и веса строк (спринт 33).

## 3. PSI-алярм (дрейф фич)

**Симптом.** `wp_psi_max` > 0.2, уведомление «⚠️ дрейф фич»; авто-обучение
запускается по триггеру drift чаще обычного.

**Причины.**

1. **Баланс-патч Dota** — ожидаемое поведение контура CT (Гл. 10.4,
   R-02): мета сменилась, PSI-триггер для того и заведён. Ничего не
   делать; следить, что гейт пропустил модель на новых данных. При
   большом патче рассмотреть даунвейт старых матчей (A9 роадмапа).
2. **Сломался источник данных**, а не мета: например, коллектор начал
   писать нули/NaN в одну фичу (обрыв позиций → `position_advance`).
   Отличие от патча: дрейфит одна фича, а не спектр; смотреть
   `wp_feature_psi{feature=...}` по отдельности.
3. **Перенаполнение витрины** после сброса/ремиграции — распределение
   то же, но состав матчей другой. Разовый алярм, самоуспокоится.

## 4. Реплейный путь стоит (витрина при этом растёт)

**Симптом.** Уведомление «⚠️ реплейный путь стоит» (auto-train,
`REPLAY_STALL_ALERT_H`, по умолчанию 6 ч), метрика
`training_replay_path_stalled=1`. Витрина может продолжать расти —
JSON-путь маскирует проблему, поэтому у реплейного пути отдельный алерт.

**Диагностика.** `make doctor` — покажет отсутствующие топики, мёртвые
консьюмер-группы и свежесть `ReplayEvents` одним прогоном.

**Причины по частоте.**

1. **Потерянные Kafka-топики** (volume пересоздался; автосоздание
   выключено): продюсер теряет сообщения МОЛЧА. С спринта 49 recover
   сам прогоняет `make topics`, но если стек не перезапускали —
   `make topics` + перезапуск parser-svc и feature-extractor.
2. **Упавший parser-svc / extractor** — смотреть `~/manta-logs/`.
3. **Квота OpenDota** — реплейные коллекторы уснули до 00:00 UTC
   (это нормально; см. runbook 1).

---

Общий принцип: сначала `make doctor` (быстрый вердикт по данным), затем
`make recover` (безопасен на живом стеке; сам чинит топики/миграции),
потом диагностика по метрикам (`scripts/dashboard.py` — живой дашборд без
Grafana), и только затем ручные вмешательства в контейнеры/данные.
